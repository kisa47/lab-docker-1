# Docker

Docker — это платформа для контейнеризации приложений. Она позволяет упаковать приложение вместе со всеми его зависимостями, библиотеками и настройками в изолированную среду — контейнер. Такой контейнер можно запускать на любом сервере или компьютере, и он будет работать одинаково, независимо от окружения.

## Основные понятие

- **Контейнер** (*Container*)
Экземпляр контейнера, представляющий собой изолированную среду выполнения приложения, созданную на основе образа. Контейнер обеспечивает изоляцию процессов, файловой системы, сети и ресурсов.

- **Образ** (*Image*)
Неизменяемый шаблон, содержащий файловую систему, зависимости, библиотеки, конфигурационные файлы и инструкции для запуска приложения. Образы строятся послойно и хранятся в реестре.

- **Dockerfile**
Текстовый файл, содержащий набор инструкций для автоматизации сборки образа Docker. Он описывает окружение, зависимости и команды, необходимые для запуска приложения в контейнере. Если не задан при сборке **Образа**, используется файл с именем **Dockerfile** в текущей директории.

- **Контекст сборки** (*build context*)
Набор файлов и директорий, которые передаются **Docker** при сборке **Образа** (*Image*).  

- **Слой образа** (*Image Layer*)
Структурная единица образа, представляющая собой набор изменений файловой системы. Слои объединяются с помощью *Union File System* и обеспечивают эффективное хранение и повторное использование.

- **dockerignore**
Текстовый файл, который определяет, какие файлы и директории следует исключить (при использовании инструкций **ADD** и **COPY**)при сборке Docker-образа. Он работает аналогично **.gitignore**, но предназначен для оптимизации сборки контейнеров.  

- **Том** (*Volume*)
Механизм постоянного хранения данных, отделённый от жизненного цикла контейнера. Тома используются для хранения баз данных, логов и других данных, требующих сохранности между перезапусками.

- **Реестр** (*Registry*)
Хранилище **образов** (*Images*), которое позволяет загружать (*push*), скачивать (*pull*) **образы** (*Images*). Для работы с **реестром** (*Registry*) может потребоваться авторизация. По умолчанию (если не задан или не переопределён системными настройками) используется общедоступный реестр [Docker Hub](https://hub.docker.com/).

- **Репозиторий** (*Repository*)
Коллекция **образов** (*Images*) в **реестре** (*Registry*) под одним именем. Обычно содержит несколько версий **образов** (*Images*).

- **Тег** ( *Tag*)
Метка, указывающая на конкретную версию **образа** (*Images*) внутри репозитория. По умолчанию (если не задан) - *latest*.

- **Имя образа** (*Image Name*)
Имя образа — это обозначение, которое состоит из **реестра**, **репозитория** и **тега**. Обозначанется как ```[registry/]repository[:tag]```. При этом обязательным является только **Репозиторий** (*Repository*). Если не задано имя образа при сборке, то присваивается одно из имён разработчиков **Docker**.

## Dockerfile

Текстовый файл, содержащий набор инструкций для автоматизации сборки образа Docker. Он описывает окружение, зависимости и команды, необходимые для запуска приложения в контейнере.  

Сборка **образа** (*Image*) осуществляется "по шагам" описанным в **Dockerfile** сверху вниз. Результатом выполнения такого "шага" является **слой образа** (*Image Layer*).  

Каждый **слой образа** (*Image Layer*) сохраняется в системе, а итоговый **образа** (*Image*) является последовательным объединением этих **слойов образа** (*Image Layer*).  

При получении или отправки **образа** (*Image*) происходит передача всех **слойов образа** (*Image Layer*).  

Пример содержания:

```dockerfile
# Базовый образ с Python 3.13 (Может иметь свои слои образа)
FROM python:3.13-slim

# Рабочая директория внутри контейнера (первый слой образа)
WORKDIR /app

# Копируем файл с зависимостями (второй слой образа)
COPY requirements.txt .

# Устанавливаем зависимости (третий слой образа)
RUN pip install --no-cache-dir -r requirements.txt

# Копируем исходный код приложения (четвертый слой образа)
# При разработке приложения код меняется чаще зависимостей
# поэтому копирование исходного слоя стоит делать после
# установки зависимостей
COPY . .

# Переменная окружения (пятый слой образа)
ENV FLASK_ENV=production

# Порт, который будет использоваться (шестой слой образа)
EXPOSE 5000

# Команда для запуска приложения (седьмой слой образа)
CMD ["python", "app.py"]
```

## Рекомендации по написанию Dockerfile

- Минимизируйте количество слоев (объединяйте связанные команды в инструкции **RUN** через *&&* или *;*).  
- Используйте **.dockerignore**, чтобы исключить ненужные файлы.  
- Размещайте редко изменяющиеся инструкции (например, ```RUN apt-get update && apt-get install -y ...```) выше часто меняющихся (например, ```COPY . .```).  
- Используйте, при возможности, легковесные базовые образы (например, **alpine**).

## Основные инструкции Dockerfile

| **Инструкция** | **Описание** | **Пример использования** |
| :--- | :--- | :--- |
| **FROM** | Задает имя базового образа, от которого наследуется текущий. | `FROM python:3.11-slim` |
| **SHELL** | Задает интерпретатор по умолчанию для инструкций **RUN**, **CMD** и **ENTRYPOINT**. | `SHELL ["/bin/bash", "-c"]` |
| **RUN** | Выполняет команды в процессе сборки образа (например, установка пакетов). | `RUN apt-get update && apt-get install -y curl` |
| **COPY** | Копирует файлы из локальной системы в файловую систему образа. | `COPY . /app` |
| **ADD** | Аналогично `COPY`, но поддерживает распаковку архивов и загрузку по URL. | `ADD app.tar.gz /app/` |
| **WORKDIR** | Устанавливает рабочую директорию для последующих инструкций. | `WORKDIR /app` |
| **ENV** | Определяет переменные окружения внутри контейнера. | `ENV FLASK_ENV=production` |
| **EXPOSE** | Объявляет порты, которые контейнер будет использовать (документация). | `EXPOSE 8080` |
| **VOLUME** | Создает точку монтирования для постоянных данных. | `VOLUME /data` |
| **LABEL** | Добавляет метаданные к образу (автор, версия, описание). | `LABEL maintainer="dev@example.com"` |
| **CMD** | Задает команду по умолчанию, выполняемую при запуске контейнера. Может быть переопределена при запуске. | `CMD ["python", "app.py"]` |
| **ENTRYPOINT** | Определяет исполняемый файл, который всегда запускается при старте контейнера. Аргументы из **CMD** или командной строки добавляются к нему. | `ENTRYPOINT ["python", "app.py"]`<br>`CMD ["--port", "8000"]` |

## Работа с Образами (Images)

- Просмотр списка **образов** (*Images*) на хосте:
```bash
docker images # вывод всех образов
docker images repository # выводит все образы (все версии) образов репозитория
```
- Сборка **образа** (*Image*) из **Dockerfile**:
```bash
docker build . ## Простая сборка c контекстом в текущей директории
docker build ./src # Задан путь до контекста сборки
docker build . -t my-app:v1.1.1 # Задано имя собираемого образа (реестр используется по умолчанию)
docker build . -t my-app # Задано имя собираемого образа без тега (используется my-app:latest)
docker build . -f ./Dockerfile.alpine # Задан путь до Dockerfile
docker build ./src -f ./Dockerfile.alpine -t my-app:v1.1.1
```
- Получение образа из **Реестра** (*Registry*):
```bash
docker pull my-app # Получение образа my-app из реестр по умолчанию с тегом latest
docker pull my-registry.org/my-app:v1.1.1 # Получение образа my-app из реестра my-registry.org с тегом v1.1.1
```
- Отправка образа в **Реестр** (*Registry*):
```bash
docker push my-app # Отправка образа my-app в реестр по умолчанию с тегом latest
docker push my-registry.org/my-app:v1.1.1 # Отправка образа my-app в реестр my-registry.org с тегом v1.1.1
```
- Удаление образа
```bash
docker rmi my-app # удаление образа my-app с тегом latest
docker rmi my-app:v1.1.1 # удаление образа my-app с тегом v1.1.1
```

Вот структурированный справочник по основным операциям с контейнерами Docker, оформленный в том же стиле, что и ваш пример с образами.

## Работа с Контейнерами (Containers)

### Просмотр состояния и списка контейнеров
```bash
docker ps # Показывает только запущенные контейнеры
docker ps -a # Показывает все контейнеры (в том числе остановленные)
docker ps -l # Показывает последний созданный контейнер
docker ps -f status=exited # Фильтрация по статусу (например, только остановленные)
```

### Запуск контейнера
```bash
docker run nginx # Запуск контейнера из образа nginx (в интерактивном режиме)
docker run -d nginx # Запуск в фоновом режиме (detached)
docker run -d -p 8080:80 nginx # Проброс порта: 8080 (хост) -> 80 (контейнер)
docker run --name web-server -d nginx # Задание имени контейнера
docker run -e ENV_VAR=value -d my-app # Передача переменной окружения
docker run --rm -it ubuntu bash # Запуск с автоматическим удалением после выхода
```

### Остановка и запуск
```bash
docker stop <container_id> # Остановка контейнера (корректно, через SIGTERM)
docker start <container_id> # Запуск остановленного контейнера
docker restart <container_id> # Перезапуск контейнера
```

### Удаление контейнеров
```bash
docker rm <container_id> # Удаление остановленного контейнера
docker rm -f <container_id> # Принудительное удаление работающего контейнера
```

### Взаимодействие с работающим контейнером
```bash
docker exec -it <container_id> bash # Вход в интерактивную оболочку контейнера
docker logs <container_id> # Просмотр логов контейнера (по умолчанию stdout/stderr)
docker logs -f <container_id> # Следить за логами в реальном времени (как tail -f)
docker top <container_id> # Список процессов, запущенных внутри контейнера
```

### Получение информации и инспекция
```bash
docker inspect <container_id> # Детальная информация о контейнере в формате JSON
docker port <container_id> # Показывает проброшенные порты контейнера
```

### Полезные флаги (Options)
*   `-d` или `--detach`: Запуск в фоновом режиме.
*   `-p` или `--publish`: Проброс портов (`-p host_port:container_port`).
*   `--name`: Задание имени контейнера.
*   `-e` или `--env`: Установка переменных окружения.
*   `-v` или `--volume`: Монтирование томов (`-v /host/path:/container/path`).
*   `--rm`: Автоматическое удаление контейнера при остановке.
